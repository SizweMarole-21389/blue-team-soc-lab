## Overview

In this project I simulated a network reconnaissance attack using Nmap from the Kali Linux attacker machine against the Ubuntu 24.04 victim on AWS EC2. I then detected the scan activity in Splunk by analysing authentication and syslog data for patterns consistent with port and service scanning.

Network reconnaissance is almost always the first phase of any attack. Detecting it early gives defenders a significant advantage, as it signals that a threat actor is mapping the environment before launching a targeted attack.

## Attack Simulation

I performed a combined SYN scan and service version detection scan from the Kali machine targeting the victim's public IP address. This is the most common type of reconnaissance an attacker would run after identifying a target host.

**Nmap command used:**

```bash
nmap -sV -sS 13.246.220.248
```

**Flags explained:**

| Flag | Meaning |
|------|---------|
| -sS | SYN (stealth) scan - sends SYN packets and records responses |
| -sV | Service version detection - identifies what is running on open ports |

The scan probed all 1000 default ports and returned open services, versions, and the operating system fingerprint.

**Additional targeted scan:**

```bash
nmap -sV -p 22,80,443,3306 13.246.220.248
```

---

## What Nmap Leaves Behind

Even though a SYN scan does not complete the TCP handshake, it still generates evidence in multiple places:

1. **auth.log** - Connection attempts to SSH (port 22) appear as incomplete or refused connection entries.
2. **syslog** - The kernel's TCP stack may log connection resets from the rapid port probing.
3. **Network traffic** - A burst of SYN packets from a single source IP to many destination ports in a very short time window is the definitive signature of a port scan.
4. **ufw log** - If UFW (firewall) is enabled, blocked connection attempts from the scanning IP are logged.

## Splunk Detection

### Query 1 - Connection Spike from Single IP

This query detects a high volume of connection-related log entries from a single source:

```spl
index=* source="/var/log/auth.log"
| rex "from (?P<src_ip>\d+\.\d+\.\d+\.\d+)"
| stats count by src_ip
| where count > 20
| sort -count
```

### Query 2 - Rapid Port Knock Pattern in Syslog

```spl
index=* source="/var/log/syslog"
| rex "(?P<src_ip>\d+\.\d+\.\d+\.\d+)"
| stats dc(src_ip) as unique_sources, count by src_ip
| where count > 50
| sort -count
```

### Query 3 - UFW Blocked Connections (if UFW is active)

```spl
index=* source="/var/log/ufw.log" "BLOCK"
| rex "SRC=(?P<src_ip>\d+\.\d+\.\d+\.\d+)"
| rex "DPT=(?P<dst_port>\d+)"
| stats dc(dst_port) as ports_probed by src_ip
| where ports_probed > 15
| sort -ports_probed
```

A single IP probing more than 15 distinct destination ports in a short window is a strong indicator of automated scanning.

## Network-Level Evidence

During the simulation I also ran a tcpdump capture on the victim machine to capture the raw SYN packets generated by Nmap. The capture file confirmed:

- A burst of SYN packets from the Kali IP to the victim
- Rapid sequential probing of port numbers 1 through 1000
- RST/ACK responses from closed ports
- SYN-ACK from port 22 (open)

This PCAP analysis is covered in detail in [Project 04 - PCAP Network Forensics](../04-pcap-network-forensics/README.md).

## Response Actions

1. Identify the scanning IP and check it against threat intelligence (AbuseIPDB, Shodan).
2. Block the IP at the security group level in AWS.
3. Review what services were discovered - if sensitive ports are open, remediate immediately.
4. Check for any subsequent attack activity from the same IP (brute force, exploit attempts).
5. Document the discovery scope (which ports were probed, which were open) for the IR report.

---

## Key Takeaways

- Nmap scans generate a high volume of connection attempts in a very short time window - this is detectable even without a dedicated network sensor.
- Correlating auth.log, syslog, and firewall logs together gives a much stronger detection signal than any single source alone.
- In cloud environments like AWS, enabling VPC Flow Logs provides an additional detection layer that captures all connection attempts regardless of whether they reach the OS.
- Blocking the attacker IP after recon detection may prevent the next phase of the attack before it starts.
